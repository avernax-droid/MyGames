1. VISÃO GERAL
Nome: MyGames
Objetivo: Sistema de recompra (Buyback) de jogos e consoles usados para um cliente de e-commerce. O sistema capta o lead, permite ao usuário selecionar produtos, indicar o estado de conservação (perícia), calcula a depreciação e exibe uma oferta.
Fluxo do Usuário (Stepper):
Boas-vindas/Canal > Seleção de Produto > Perícia/Fotos > Resultado da Oferta > Resumo do Lote > Identificação > Sucesso/Protocolo.

2. AMBIENTE E TECNOLOGIAS
Linguagem/Framework: Python com Flask.
Banco de Dados: MySQL (Banco: mygames_dev).
Infraestrutura: Docker (planejado/em uso), Ngrok (para túnel de testes web/mobile).
Ambiente de Desenvolvimento: Host em Windows 11 utilizando VS Code (Tema Dark Mode) com terminal integrado em Linux - Ubuntu.
Metodologia: "Design First" (As interfaces visuais, regras de UX e estruturas de banco são definidas antes da codificação).

engine.py [MÓDULO PRINCIPAL CORE]: É o coração do sistema. Gerencia toda a camada de persistência (MySQL), cálculos matemáticos de perícia (depreciação e multiplicadores regionais), integração com APIs externas (IBGE), envio de e-mails automatizados e gestão segura de leads via UPSERT (validação por CPF).

server.py: Servidor web Flask. Atua como controller. Gerencia as rotas, o fluxo da session do usuário, processa uploads de arquivos e atende os endpoints de Handoff Remoto (recebendo mídias via mobile com prefixação UUID e respondendo status via polling).

Diretório /templates:
base.html: Estrutura padrão de layout.
boas_vindas.html: Landing page, captura canal de aquisição e cidade (integração IBGE).
produto.html: Catálogo de seleção.
pericia.html: Formulário de estado de conservação, comentários e interface dividida em abas para upload local via PC (com feedback visual de arquivos) ou handoff remoto gerando QR Code dinâmico para o smartphone. Controla a sincronização AJAX de forma isolada para evitar falhas ao alternar abas.
upload_mobile.html: Interface mobile ultra leve acessada pelo QR Code. Acessa a câmera nativa do celular, acumula múltiplas fotos em memória (array JS) e envia os arquivos para o Flask via API.
resultado.html: Exibe a cotação. Possui lógica condicional para multiplicador de região e itens "Sob Consulta".
resumo_lote.html: Carrinho/resumo dos itens avaliados.
cadastro_complementar.html: Formulário final de captura de lead completo.
sucesso.html: Geração do protocolo final.

Diretório /static:
/css/style.css, /js/ux-helper.js (Modais e funções auxiliares).
/uploads/pericia: Destino das fotos enviadas pelos usuários.

5. REGRAS DE NEGÓCIO E MECÂNICAS IMPORTANTES
Precificação Regional: O front-end e o e-mail exibem alertas visuais ("✨ Bônus Regional Aplicado!") e a ancoragem do preço original riscado se a cidade tiver multiplicador diferente de 1.0 na tabela regioes_atendimento.
Itens "Outros" / Sob Consulta: Produtos com prefixo cat_ ou outro_cat_ ativam a flag is_outros. O preço não é calculado na hora; o sistema e o e-mail avisam sobre a avaliação humana em até 24h úteis.
Política de Avaliação: O e-mail final contém uma cláusula jurídica informando que a oferta está sujeita à avaliação técnica presencial.
Modal de Descarte: O usuário pode ejetar itens do lote em resultado.html sem recarregar a tela (AJAX).
Gestão de Leads (UPSERT): Para evitar duplicação de clientes na base, a função salvar_lead realiza um UPSERT manual. Se o CPF fornecido já existir no banco, o sistema faz um UPDATE nos dados de contato/endereço em vez de gerar um novo ID.
Handoff Mobile (QR Code): Se o usuário acessar via desktop e desejar enviar fotos do celular, o sistema gera uma sessão com Token UUID e exibe um QR Code. O celular envia as fotos acumuladas em memória, e o servidor as salva com um prefixo UUID para evitar colisão de nomes. A tela do desktop atualiza automaticamente via polling de JavaScript (AJAX), unificando a experiência de envio de mídias sem transferência manual de arquivos.
