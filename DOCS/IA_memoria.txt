1. VISÃO GERAL
Nome: MyGames
Objetivo: Sistema de recompra (Buyback) de jogos e consoles usados para um cliente de e-commerce.
O sistema capta o lead, permite ao usuário selecionar produtos, indicar o estado de conservação (perícia), calcula a depreciação e exibe uma oferta.

Fluxo do Usuário (Stepper):
Boas-vindas/Canal > Seleção de Produto > Perícia/Fotos > Resultado da Oferta > Resumo do Lote > Termos da Oferta/Política > Identificação > Sucesso/Protocolo.

2. AMBIENTE E TECNOLOGIAS
Linguagem/Framework: Python com Flask.
Banco de Dados: MySQL (Banco: mygames_dev).
Infraestrutura: Docker (planejado/em uso), Ngrok (para túnel de testes web/mobile).
Ambiente de Desenvolvimento: Host em Windows 11 utilizando VS Code (Tema Dark Mode) com terminal integrado em Linux - Ubuntu.
Metodologia: "Design First" (As interfaces visuais, regras de UX e estruturas de banco são definidas antes da codificação).

engine.py [MÓDULO PRINCIPAL CORE]: É o coração do sistema.
Gerencia toda a camada de persistência (MySQL), cálculos matemáticos de perícia (depreciação e multiplicadores regionais), integração com APIs externas (IBGE), envio de e-mails automatizados, gestão de UUIDs e gestão segura de leads via UPSERT (validação por CPF). Persiste também logs de feedback de recusa de ofertas.

server.py: Servidor web Flask. Atua como controller. Gerencia as rotas, o fluxo da session do usuário, processa uploads de arquivos, processa ações de descarte via AJAX e atende os endpoints de Handoff Remoto (recebendo mídias via mobile com prefixação UUID e respondendo status via polling).

Diretório /templates:
base.html: Estrutura padrão de layout.
boas_vindas.html: Landing page, captura canal de aquisição e cidade (integração IBGE).
produto.html: Catálogo de seleção.
pericia.html: Formulário de estado de conservação, comentários e interface dividida em abas para upload local via PC ou handoff remoto gerando QR Code dinâmico.
upload_mobile.html: Interface mobile acessada pelo QR Code. Acessa a câmera nativa, acumula fotos em memória e envia para o Flask.
resultado.html: Exibe a cotação. Possui lógica condicional para multiplicador de região e itens "Sob Consulta".
resumo_lote.html: Carrinho/resumo dos itens avaliados. Estruturado em tabela visual (com coluna QTD) com blindagem CSS (table-layout: fixed e word-break) contra textos contínuos que quebrem o layout.
termos_oferta.html: Interface de exibição da política da empresa pós-oferta. Utiliza checkboxes de exclusividade mútua para forçar o aceite obrigatório antes de avançar, ou permite a recusa revelando uma área inline opcional de feedback (sem modal), descartando o lote em seguida.
cadastro_complementar.html: Formulário final de captura de lead completo.
sucesso.html: Geração do protocolo final.

Diretório /static:
/css/style.css, /js/ux-helper.js (Modais, funções auxiliares, busca de CEP e chamadas AJAX de descarte).
/uploads/pericia: Destino das fotos enviadas pelos usuários.

Detalhe da tabela feedbacks_recusa

1. user_agent (Agente de Usuário)
O que é: É uma string (uma linha de texto) que o navegador envia silenciosamente no cabeçalho (header) de cada requisição 
HTTP. Ela serve para o navegador se "apresentar" ao servidor.
O que ele revela: Ele informa qual é o navegador (Chrome, Safari, Firefox, Edge), a versão desse navegador e o sistema 
operacional / dispositivo da pessoa (Windows 11, Mac, iOS, Android).
Por que é estratégico guardar isso nas recusas: Diagnóstico de Interface e Comportamento do Cliente.

2. canal_aquisicao
Registra a porta de entrada do usuário, ou seja, de onde essa pessoa veio antes de iniciar a avaliação dos 
jogos (exemplo: Instagram Ads, Google Orgânico, Indicação de Parceiros, Link do YouTube, etc.).
Ver a tabela canais_aquisicao. Estratégico para calcular a qualidade do lead e otimizar campanhas de marketing.

3. ip_origem (Endereço IP)
O que é: É o endereço numérico de rede de onde a conexão daquele usuário partiu. 
Por que é estratégico guardar isso nas recusas: Segurança, bloqueio de robôs (antifraude) e validação de coerência geográfica versus a cidade informada pelo usuário.
Nota Técnica de Infraestrutura: Devido ao uso do Ngrok/Docker, o servidor lê o cabeçalho request.headers.get('X-Forwarded-For') para capturar o IP real.

4. sessao_uuid (Identificador Único)
O que é: Universally Unique Identifier (Identificador Único Universal). Um código alfanumérico longo gerado para a sessão.
Por que é estratégico guardar isso nas recusas: Permite a identificação e o isolamento de recusas de usuários que ainda não chegaram à tela de cadastro de dados (CPF/E-mail). Mantém a padronização arquitetônica de rastreabilidade (já usada no Handoff Mobile do sistema) e atua como ferramenta chave para auditoria de logs no servidor.

5. REGRAS DE NEGÓCIO E MECÂNICAS IMPORTANTES
Precificação Regional: O front-end e o e-mail exibem a ancoragem do preço ajustado se a cidade tiver multiplicador diferente de 1.0 na tabela regioes_atendimento.
Itens "Outros" / Sob Consulta: Produtos com prefixo cat_ ou outro_cat_ ativam a flag is_outros. O preço não é calculado na hora; o sistema e o e-mail avisam sobre a avaliação humana em até 24h úteis.
Política de Avaliação: O e-mail final contém o valor total do lote destacado no topo e uma cláusula jurídica informando que a oferta está sujeita à avaliação técnica presencial.
Modal de Descarte: O usuário pode ejetar itens individuais do lote em resumo_lote.html sem recarregar a tela (AJAX) ou descartar o lote inteiro.
Gestão de Leads (UPSERT): Para evitar duplicação de clientes na base, a função salvar_lead realiza um UPSERT manual. Se o CPF fornecido já existir no banco, o sistema faz um UPDATE nos dados de contato/endereço em vez de gerar um novo ID.
Handoff Mobile (QR Code): Se o usuário acessar via desktop e desejar enviar fotos do celular, o sistema gera uma sessão com Token UUID e exibe um QR Code. O celular envia as fotos acumuladas em memória, e o servidor as salva com um prefixo UUID. A tela do desktop atualiza automaticamente via polling de JavaScript (AJAX).
Política de Aceite e Recusa de Oferta (Barreira de Validação): Após visualizar a cotação e conferir os itens em resumo_lote.html, o usuário deve passar pela tela de termos da oferta (termos_oferta.html). O avanço para a identificação só ocorre se o checkbox de aceite for explicitamente marcado. Caso marque o checkbox de recusa, uma área inline opcional recolhe o feedback textual e dispara uma persistência assíncrona na tabela feedbacks_recusa, capturando o estado do carrinho (JSON), valor global recusado, identificador UUID, origem e metadados, executando a limpeza da sessão e redirecionando para a página inicial logo em seguida.
