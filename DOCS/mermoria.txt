O Que Estávamos Testando (Escopo Mobile V3.0)
Estamos implementando a Otimização Mobile para o sistema MyGames, focada em converter layouts rígidos (tabelas e grids de desktop) para interfaces fluidas e focadas em conversão (Mobile First).

Arquitetura de Roteamento: Uso de render_smart_template para alternar dinamicamente entre templates /desktop e /mobile.

UX Mobile: Bloqueio de auto-zoom em inputs, botões de ação empilhados verticalmente e touch-targets amplos para facilitar o uso com uma mão.

Perícia e Upload: Implementação de upload nativo via capture="environment" (Câmera) e multiple (Galeria), com galeria de miniaturas em tempo real via JavaScript (FileReader e DataTransfer).

Fluxo de Finalização: Implementação de Cards Acionáveis em Termos de Oferta e um fluxo de cadastro simplificado (com campos numéricos nativos e busca de CEP via ViaCEP).

Diagnóstico de Erros e Como Corrigir
1. Erro de Rota/Caminho Duplicado nas Imagens (Log: 404 Not Found)
Problema: O log mostrava GET /media/pericia/media/pericia/.... Isso ocorre porque a função que serve as imagens está recebendo um caminho já prefixado duas vezes.

Correção: No seu templates/pericia.html (ou no js que processa o polling do QR Code), identifique onde o src da imagem é montado.

Se a API retorna apenas o nome do arquivo, mantenha o prefixo /media/pericia/.

Se a API retorna o caminho completo, remova o prefixo manual do JavaScript.

Dica: Use console.log(fotoUrl) no seu JS para inspecionar exatamente o que está chegando antes de atribuir ao src.

2. Erro de Sintaxe Jinja2 (jinja2.exceptions.TemplateSyntaxError)
Problema: O Flask não conseguiu compilar o template porque uma tag de bloco estava aberta mas não foi fechada ({% block %} sem o respectivo {% endblock %}).

Correção: Verifique sempre a paridade das tags:

Cada {% block content %} deve ter um {% endblock %} logo após o conteúdo daquela seção e antes de abrir um novo bloco (como o {% block scripts %}).

3. UX do Fluxo de Upload (Upload PC vs. QR Code)
Problema: Botões concorrentes não limpavam o estado visual (ex: quadro do QR Code persistindo após o clique em Upload PC).

Correção: Certifique-se de que a função de clique do "Upload do Computador" chame explicitamente a ocultação do elemento area-upload-celular (style.display = 'none') e limpe o clearInterval(pollingInterval) para evitar que o código tente buscar fotos via rede quando o usuário decidiu fazer upload direto.

Próximos Passos Sugeridos para Documentação
Validação de API: Certifique-se de que a rota /api/status-upload/<token> esteja retornando um JSON limpo contendo apenas a lista de arquivos (ex: {"status": "concluido", "fotos": ["foto1.jpg", "foto2.jpg"]}).

Padronização CSS: Todos os novos componentes mobile utilizam a variável --cor-destaque e o tom #d48806 para ênfases, garantindo a identidade visual.

27/06

Esta é uma mudança estratégica importante para o MyGames. Passar de um modelo de "perícia de lote" para "perícia de itens" (granular) aumenta a precisão e a profissionalização do atendimento ao cliente.

Abaixo, apresento o plano detalhado de implementação para a sua documentação.

1. Mudanças na Estrutura do Banco de Dados
Precisamos evoluir a tabela itens_periciados para suportar o estado individual de cada item, permitindo diferenciar o que foi declarado do que foi efetivamente recebido.

A. Comandos SQL Sugeridos
Execute estes comandos para preparar a tabela de itens:
-- Adiciona campos para controle granular
ALTER TABLE itens_periciados 
ADD COLUMN status_item VARCHAR(20) DEFAULT 'pendente', -- pendente, aprovado, reprovado, ausente
ADD COLUMN qtd_declarada INT DEFAULT 1,
ADD COLUMN qtd_recebida INT DEFAULT 0,
ADD COLUMN motivo_recusa TEXT NULL,
ADD COLUMN data_pericia TIMESTAMP DEFAULT CURRENT_TIMESTAMP;

2. Módulos e Impactos no Código
A lógica de processamento deixará de ser um "disparo único" no protocolo e passará a ser uma atualização por linha (item).

A. admin_server.py (Lógica de Backend)
Nova Rota de Atualização: Criar uma rota (ex: /api/pericia/update-item) que recebe apenas o item_id, status e qtd_recebida.

Cálculo Automático: Após cada atualização de item, o servidor deve verificar o status geral do protocolo:

Se todos os itens estiverem com status final (aprovado/reprovado/ausente) ➔ Alterar status do protocolo para "Periciado".

Trigger de Comunicação: Se um item for marcado como "ausente", a lógica deve sinalizar ao sistema que um e-mail de divergência deve ser 
preparado ou enviado.

B. templates/pericia_esteira.html (Interface do Operador)
Transformação em Formulário: Os itens que hoje estão em uma lista (ou acordeão) passarão a ser campos de entrada (<input> para quantidade, 
<select> para status e <textarea> para motivo).

Feedback Visual: Adicionar cores dinâmicas para cada linha:

Verde (Aprovado), Vermelho (Reprovado), Amarelo (Ausente/Divergente).

Botão de Salvar Individual: Cada linha deve ter um botão "Salvar Item" (via AJAX) para que o operador não precise salvar o lote inteiro de 
uma vez, evitando perda de dados.

3. Fluxo de Trabalho (Workflow) Proposto
Para documentar o comportamento do sistema, siga este fluxo:

Entrada: O protocolo chega no sistema. O operador abre a pericia_esteira.html.

Conferência: Para cada item, o operador insere a qtd_recebida.

Caso: Operador insere 0 em um item. Sistema marca automaticamente como Ausente.

Avaliação: Para cada item, o operador seleciona Aprovado ou Reprovado.

Caso: Reprovado ➔ Abrir campo motivo_recusa (obrigatório).

Notificação: Ao finalizar a perícia do último item:

O sistema envia um e-mail ao cliente contendo:

Lista dos itens aprovados e valores.

Lista dos itens reprovados (com motivos).

Lista dos itens ausentes (com solicitação de confirmação).

4. Próximos Passos para Desenvolvimento
Para não se perder na implementação, recomendo seguir esta ordem:

Migração do Banco: Execute o ALTER TABLE no seu ambiente de desenvolvimento.

Ajuste do Controller: Crie a rota api/pericia/update-item em admin_server.py. Teste via Postman ou Insomnia para garantir que o banco está 
gravando o status de cada item corretamente.

Refatoração do Front-end: Atualize o templates/pericia_esteira.html para exibir os campos de edição e o botão de salvar.

Implementação do E-mail: Crie a lógica de disparo de e-mail baseada no resultado final do lote após a conferência de todos os itens.

Este plano cobre a necessidade de granularidade e o controle de divergências (itens ausentes). Se precisar que eu escreva o código da nova 
rota de backend ou o HTML para a nova interface da esteira, é só me pedir!

1. Correção na Persistência de Dados Granulares (Perícia)

Problema: O sistema realizava o update do status de recebimento dos itens, mas os valores finais (valor_final_pix) e os laudos textuais

Problema: A consulta que extraía os dados para montar o e-mail não buscava o valor pós-perícia, forçando a exibição do valor original 
de catálogo para todos os itens.

Solução: Atualização da instrução SQL na função obter_itens_protocolo (em admin_engine.py) para realizar o SELECT trazendo as colunas 
i.valor_final_pix e i.motivo_recusa.

3. Injeção Global da Logística Reversa (Rastreio)

admin_server.py). O servidor agora injeta as variáveis codigo_rastreio e status_rastreio tanto no gatilho da Triagem (Regra 1) quanto 
no gatilho do Laudo Final (Regra 2).
