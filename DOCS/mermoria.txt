MEMÓRIA DE CONTEXTO: PROJETO MYGAMES (V1.7)
1. O QUE É O PROJETO
O MyGames é um sistema de logística reversa e recompra de produtos gamers (consoles e jogos usados). 
Ele funciona como uma "esteira" que transforma a intenção de venda do cliente em um protocolo de envio para a loja.

2. COMO O SISTEMA É ACESSADO (3 PORTAS)
Direto: Link próprio do MyGames[cite: 7].

Redirecionamento: Usuários vindos do site principal (lojatrocagame.com.br)[cite: 7].

API/Chatbot: Interface para receber dados de sistemas de terceiros (a "Caixa Preta" mencionada)[cite: 7].

3. O FLUXO DE TELAS (INTERPRETAÇÃO DO BENCHMARK)
Interpretei que a aplicação deve ser estritamente visual e intuitiva[cite: 7]:

Etapa 1 (Captação): O sistema primeiro "garante" o lead (Nome, WhatsApp, E-mail)[cite: 7].

Etapa 2 (A Busca Inteligente): O usuário busca o que quer vender. Ao digitar, abre-se um box de produtos afins. Existem dicas de 
busca para evitar erros[cite: 7].

Etapa 3 (A Perícia Visual): O sistema exibe a foto oficial do item escolhido. O usuário seleciona a quantidade e marca as condições 
em um checklist dinâmico (caixa, manual, riscos)[cite: 7].

Etapa 4 (Carrinho): Permite adicionar vários itens antes de gerar o protocolo final[cite: 7].

4. REGRAS DE NEGÓCIO E DADOS
Precificação: O motor (engine.py) usa a planilha "2025" para calcular dois valores: PIX (dinheiro) e Crédito (troca na loja)[cite: 7].

Logística: O protocolo gera um WhatsApp imediato. Após o aceite da cotação por e-mail, o sistema coleta dados sensíveis (CPF, PIX, 
Endereço) e gera o código de frete reverso baseado no número de caixas[cite: 7].

Prioridade: Este projeto é a prioridade número 1, o MyMiles está em segundo plano por enquanto[cite: 2].

automatizar o fluxo que vai desde o interesse do usuário em vender um item até o recebimento desse produto pela loja, com precificação dinâmica e inteligente.

Resumo de Contexto: Modelo de Integração MyGames
1. O Problema Identificado (Sistema)
O computador apresentou instabilidade (apps fechando) devido a um APPCRASH no driver de vídeo Intel (igfxCUIService.exe). Foram executados sfc /scannow (reparou arquivos) e dism /restorehealth para estabilizar o Windows 11 e proteger o ambiente de desenvolvimento (VS Code, Docker, MySQL).

2. A Solução Proposta: Modelo 2 (Webhook Push + Handover)
Este modelo foi escolhido para integrar o Agente IA (WhatsApp/Caixa Preta) ao site MyGames (Flask/Python).

O Fluxo Técnico (Passo a Passo):

Coleta: O Chatbot coleta os dados do usuário e do console no WhatsApp.

Disparo (POST): O Chatbot faz uma requisição HTTP POST enviando um JSON para o seu servidor (server.py).

Exemplo de payload: {"cliente": "Antonio", "produto": "PS5", "condicao": "usado"}.

Processamento: Seu engine.py recebe o JSON, valida os dados, salva no MySQL e gera um ID_TRANSACAO único.

Resposta (Ack): Seu servidor responde ao POST do chatbot com esse ID (ex: {"status": "ok", "id": 1025}).

Link de Handover: O Chatbot recebe o ID e entrega ao usuário um link dinâmico: mygames.com.br/proposta/1025.

3. Por que este modelo?
Integridade: O dado não viaja pela URL (onde poderia ser editado), mas sim por uma comunicação segura entre servidores (Back-to-Back).

Experiência do Usuário: Ao clicar no link, o site já "sabe" quem é o cliente e o que ele quer vender, eliminando a redigitação.

Automação: Permite que o seu scraper.py comece a buscar preços de referência assim que o dado cai no banco, antes mesmo do usuário abrir o site.

4. Pauta para a Reunião com a Empresa de IA

Captura de Resposta: O chatbot consegue ler o ID que meu servidor retornar e injetá-lo em uma URL?

O Problema Original
O erro ocorria por dois motivos principais:

Dados aninhados: A API do IBGE não retorna um objeto simples. A estrutura é cidade -> microrregião -> mesorregião -> UF -> sigla. Se tentássemos acessar esses dados diretamente (item['microrregiao']['mesorregiao']['UF']['sigla']), o código quebrava sempre que um desses níveis estivesse ausente ou mal formado.

Performance e Filtro: Tentar filtrar cidades pelo lado do cliente (front-end) ou com muitas requisições fazia a tela parecer travada.

A Solução Implementada (O "Pulo do Gato")
Para que o boas_vindas.html funcionasse, implementamos o seguinte tratamento no server.py:

1. Acesso Seguro aos Dados (Defensive Programming)
Em vez de acessar as chaves diretamente, usamos o método .get() do dicionário encadeado com um dicionário vazio como fallback or {}. Isso impede o KeyError.

# Trecho do tratamento que salvou a busca
for m in dados:
    nome_cidade = m.get('nome', '')
    if termo in nome_cidade.lower():
        # Abertura segura nível a nível
        micro = m.get('microrregiao') or {}
        meso = micro.get('mesorregiao') or {}
        uf_obj = meso.get('UF') or {}
        uf = uf_obj.get('sigla', '??')
        
        cidades_filtradas.append({'cidade': nome_cidade, 'uf': uf})

2. Otimização da Busca no Back-End
Em vez de carregar a lista de cidades a cada caractere digitado (o que causava latência), o server.py passou a:

Receber o termo via request.args.get('q').

Validar se o termo tem pelo menos 3 caracteres (if not termo or len(termo) < 3) antes de realizar qualquer processamento. Isso reduziu drasticamente o número de requisições inúteis ao IBGE.

3. Limite de Resultados (Payload Control)
A API do IBGE retorna milhares de registros. Para não "estourar" o seu front-end, adicionamos um contador de controle dentro do loop:

if len(cidades_filtradas) >= 8:
    break

Guia Técnico: Integração API Correios (Logística Reversa)
Projeto: MyGames
Módulo Central: engine.py
Objetivo: Automatizar a geração de e-tickets (autorização de postagem) e códigos de rastreio para clientes que finalizam protocolos de venda de consoles/jogos no site, integrando diretamente com o portal CWS (Correios Web Services).

1. Credenciais e Configurações Globais (.env)
Para que a comunicação ocorra de forma segura e autenticada, a empresa precisará fornecer os dados do seu contrato 
comercial com os Correios. Essas chaves ficarão blindadas no arquivo .env do servidor:

Snippet de código
# --- INTEGRAÇÃO CORREIOS (LOGÍSTICA REVERSA) ---
CORREIOS_CNPJ=00.000.000/0000-00
CORREIOS_CONTRATO=9992345678
CORREIOS_CARTAO_POSTAGEM=0012345678
CORREIOS_USUARIO=usuario_api_fornecido_pelos_correios
CORREIOS_SENHA=senha_api_fornecida_pelos_correios
2. Estrutura no Banco de Dados (MySQL)
Seguindo a metodologia Design First, antes de processar qualquer dado, a tabela protocolos_recompra precisará ser 
expandida para armazenar as informações logísticas retornadas pela API.

Campos a serem adicionados na tabela:

e_ticket (VARCHAR): Armazena o código numérico de autorização que o cliente ditará na agência (ex: 123456789).

codigo_rastreio (VARCHAR): Armazena o código de acompanhamento (ex: QQ123456789BR).

status_logistica (VARCHAR): Controla a etapa da entrega física (ex: "Aguardando Postagem", "Em Trânsito", "Recebido no 
Laboratório").

3. Arquitetura de Software e Fluxo no engine.py
Toda a lógica de chamadas HTTP será concentrada no módulo engine.py para isolar a regra de negócio do arquivo de rotas 
server.py. O fluxo acontecerá em duas fases:

Fase A: Autenticação (Gerando o Token)
A API dos Correios exige autenticação OAuth2. O sistema fará um POST enviando CORREIOS_USUARIO e CORREIOS_SENHA para o 
endpoint de autenticação. Os Correios retornarão um Bearer Token (crachá de acesso virtual) que tem validade de algumas horas.

Fase B: O Payload de Logística Reversa
Com o Token em mãos, o engine.py montará um arquivo JSON (Payload) contendo os dados do protocolo para solicitar o 
e-ticket.

Estrutura do Payload enviado aos Correios:

Dados do Contrato: CNPJ, número do contrato e cartão de postagem.

Destinatário (MyGames): Endereço físico do laboratório de perícia da empresa.

Remetente (Cliente): Resgatado do banco de dados (Nome, CEP, Rua, Número, Cidade, UF).

Objeto: O serviço contratado (ex: SEDEX Reverso ou PAC Reverso) e o peso/dimensões estimadas da caixa 
(ex: console + controles = ~3kg).

4. O Ciclo de Vida da Integração (Passo a Passo)
Ação do Usuário: No site-cliente, o usuário clica no botão "Finalizar Protocolo". O spinner de carregamento é ativado 
na tela.

Geração Local: O engine.py cria o protocolo no banco de dados MySQL e reserva o ID (ex: Protocolo #1050).

Chamada Externa: O engine.py dispara a requisição JSON silenciosamente para os servidores dos Correios.

Resposta dos Correios: A API retorna Status 201 (Created) com o número do e-ticket e o código de rastreamento.

Gravação e Retorno: O engine.py faz um UPDATE no Protocolo #1050 salvando esses dois códigos no MySQL.

Tela de Sucesso: O spinner para de girar. O site exibe a tela de confirmação verde informando: "Tudo certo! Vá até uma 
agência dos Correios em até 7 dias e apresente o código de postagem: 123456789. O envio é por nossa conta!"

5. Exibição no Admin (Backoffice)
As informações capturadas alimentarão o Painel Administrativo que os operadores utilizam:

Tabela de Protocolos: Passará a exibir as colunas "Rastreio" e "Status Logístico" para acompanhamento rápido.

Detalhes do Protocolo (detalhes_protocolo.html): Ganhará um novo bloco ("Card") dedicado à Logística, mostrando o 
endereço de origem do cliente, o e-ticket gerado, o código de rastreio clicável (linkando para o site dos Correios) e 
a situação atual da caixa.

6. Tratamento de Erros (Fallbacks)
A internet ou a API dos Correios podem apresentar instabilidade. O engine.py deverá conter uma estrutura de try/except 
robusta:

Se a API falhar: O sistema não deve impedir o cliente de finalizar o protocolo. O protocolo será salvo no MySQL com a 
coluna e_ticket nula.

Interface do Cliente: A tela de sucesso informará: "Seu protocolo foi gerado! Em breve, nossa equipe enviará o código 
de postagem gratuita para o seu WhatsApp/E-mail."

Ação no Admin: O Backoffice mostrará um alerta visual (ex: botão vermelho "Gerar Rastreio Manualmente") para que o 
operador aperte e force o engine.py a tentar a conexão com os Correios novamente em um momento posterior.
